# ReShade source catalogues

`addons/v1/reshade.json` is the only ReShade source catalogue. Current Luma and
RenoDX v1 documents contain tool data only and do not embed these URLs. The v1
schema requires `channels.nightly` and permits `channels.stable` to be absent;
the current generator intentionally continues to publish both.

RenderPilot also ships a release-pinned copy of `addons/v1/reshade.json` as a
last-resort offline fallback. That snapshot is refreshed when the application
is released or when source-schema/security policy changes. It is not expected
to follow every CDN refresh between application releases.

## Source of truth

Stable and nightly channel URLs live only in
`scripts/lib/reshade-sources.mjs`. The generator
(`scripts/generate-reshade-manifest.mjs` → `catalogs/addons/reshade/lib/build-manifest.mjs`)
projects that module into `addons/v1/reshade.json`. Do not edit the JSON file by hand.

- **stable** — versioned `https://reshade.me/downloads/ReShade_Setup_X.Y.Z_Addon.exe`
- **nightly** — rolling `nightly.link` URLs (never version-bumped in git)

## Automation

| Workflow               | Schedule  | Purpose                                                                |
| ---------------------- | --------- | ---------------------------------------------------------------------- |
| `upstream-health.yml`  | every 12h | HEAD/GET-check committed ReShade pins (+ other pin suites)             |
| `upstream-refresh.yml` | daily     | Detect a newer stable Addon; open a PR if an installer is downloadable |

Refresh never pushes to `main` and never writes R2. Merge of the bot PR runs
the existing Publish workflow (full `pnpm check` on `main` before upload).

Detection uses GitHub tags on `crosire/reshade` plus the reshade.me homepage,
then probes every candidate newer than the pin (newest first) and selects the
first downloadable Addon. A newer tag without any live Addon is
`pending_publish` and does not rewrite sources.

The refresh workflow gates the PR on `pnpm run check:offline` (format, schema
validation, generated outputs, unit tests). Live network suites run as a
non-blocking report. PRs opened with `GITHUB_TOKEN` may not re-trigger
`pull_request` CI in the GitHub UI; validation already ran in the refresh job.

Manual commands:

```powershell
pnpm run refresh:reshade:check
pnpm run refresh:reshade:write
pnpm exec node scripts/refresh-reshade-stable.mjs --write --version=X.Y.Z
pnpm run check:upstream-health
pnpm run check:offline
```

`--write --version=X.Y.Z` skips re-scraping tags/homepage and only safety-probes
the Addon URL before rewriting (used by CI after detect).

Luma `minimum_reshade_version` is a curated compatibility floor and is **not**
updated by this automation. RenoDX catalogues are not regenerated by a ReShade
pin bump.
